# AI 驱动新范式 — 实现架构讨论

> 基于 `agents/tasks/20260128_task.txt` 的需求，先讨论方案，暂不写代码。  
> 当前 base：已打通数据流（热榜 + RSS）、邮件推送、AI 分析（LiteLLM + prompt 文件）、存储（本地/远程 SQLite）。

---

## 一、需求与现有能力对照

| 需求 | 现有能力 | 缺口 |
|------|----------|------|
| 1. 播客 RSS：检测更新 → 音频转文字 → 按 prompt 输出 | RSS 抓取、存储、新条目检测；AI 客户端；prompt 文件 | **音频下载 + 语音转文字（ASR）**；播客专用 pipeline；按「条」触发推送 |
| 2. 公众号：每日拉取 → 按 prompt 输出 | 无公众号数据源 | **公众号抓取/接口**；每日聚合 pipeline |
| 3. Twitter / HN / Kickstarter 等（可能需登录）：按话题整理 | 已有 HN RSS；无 Twitter/Kickstarter | **Twitter/Kickstarter 数据源**；登录/鉴权；话题筛选 |
| 4. 投资板块：当日信号 + 建议；港股/A 股中午、美股 23:00 推送 | 无 | **投资信号数据源**；港股/A 股/美股分区；两档推送时间 |
| 5. 指定固定话题：不限制来源，整理相关信息 | 热榜关键词、RSS、AI 分析 | **跨源检索 + 话题聚合**；可复用已有内容池 |
| 邮件三种类型 | 当前：统一时间窗口/once_per_day | **按类型拆分：播客即时 / 每日聚合 / 投资双档** |

## 二、整体架构思路

### 2.1 分层抽象

```
┌─────────────────────────────────────────────────────────────────┐
│  调度层 (Scheduler)                                               │
│  - 播客：轮询/Webhook，有更新即跑                                  │
│  - 公众号+门户：每日 1 次（可配置时间）                             │
│  - 投资：港股/A股 中午档；美股 23:00 档                             │
└─────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────┐
│  数据源层 (Sources)                                               │
│  - 播客 RSS → 音频 URL → 下载 → ASR → 文本                         │
│  - 公众号（RSS/爬虫/第三方 API）→ 正文                             │
│  - 门户/ HN / Twitter / Kickstarter → 结构化条目                   │
│  - 投资：行情/公告/大 V 文章等                                     │
└─────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────┐
│  内容池 (Content Pool)                                            │
│  - 统一存储：按 source_type + date 存；支持「已读/已推送」标记       │
│  - 可被「固定话题」检索：不限制来源，从池中按话题筛选                │
└─────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────┐
│  AI 加工层                                                        │
│  - 播客/公众号：单条内容 + 专用 prompt → 摘要/要点（config 管理）   │
│  - 投资：多源聚合 + 投资 prompt → 信号+建议                         │
│  - 固定话题：检索结果 + 话题 prompt → 整理输出                      │
│  - 模型：统一走现有 ai.* 配置，可扩展 api_base（如 siliconflow）    │
└─────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────┐
│  推送层 (Notification)                                            │
│  - 类型 1：播客 — 发现更新即发一封                                 │
│  - 类型 2：公众号+门户 — 每日一封聚合                              │
│  - 类型 3：投资 — 港股/A股 中午一封；美股 23:00 一封                │
│  - 复用现有 channel（邮件等），增加「推送类型」与「模板」            │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 配置扩展方向

- **prompt 管理**  
  - 现有：`ai_analysis_prompt.txt`、`ai_translation_prompt.txt`。  
  - 扩展：按「场景」拆文件或 config 段，例如：  
    - `prompts/podcast_summary.txt`  
    - `prompts/wechat_digest.txt`  
    - `prompts/investment_signal.txt`  
    - `prompts/topic_brief.txt`  
  - 或在 `config.yaml` 里用 `prompt_file: "xxx"` 按 source_type / report_type 绑定。

- **AI 接口**  
  - 继续用现有 `ai.model` / `ai.api_base` / `ai.api_key`。  
  - 若使用 siliconflow 等：`api_base` + `model` 指向对应端点即可，无需大改。

- **推送类型与时间**  
  - 在 `notification` 下增加「推送类型」与时间策略，例如：  
    - `podcast`: 即时，不绑时间窗口  
    - `daily_digest`: 每日一次，时间可配置  
    - `investment_cn`: 中午档  
    - `investment_us`: 23:00 档

## 三、按需求拆解的实现要点

### 3.1 播客 RSS（需求 1）

- **数据**：沿用现有 RSS 解析，拿到每集的 `enclosure`（音频 URL）。
- **新增**：  
  - 音频下载（临时文件或内存流）。  
  - 语音转文字：可选方案 — 用现有 AI 厂商的 ASR API（若 siliconflow 等提供）、或 Whisper 本地/API、或第三方 ASR 服务。  
- **逻辑**：  
  - 仅对「新条目」做：拉 RSS → 发现新 episode → 下载 → ASR → 用「播客专用 prompt」生成摘要/要点 → 写入内容池并打标记「已处理」。  
- **推送**：该播客有 1 条新处理结果就发一封（类型 1），不参与「每日聚合」时间窗口。

**可选**：播客 RSS 在 config 里与普通 RSS 区分，例如 `rss.podcast_feeds`，或用 `type: podcast` 标记，以便只对播客走 ASR+pipeline。

### 3.2 公众号（需求 2）

- **数据源**：公众号无官方 RSS，常见做法 — 第三方 RSS 服务、爬虫（易失效）、或付费 API。建议先支持「可配置的 RSS 形态公众号源」（若有第三方转 RSS）或 1～2 个明确数据源，再扩展。
- **逻辑**：  
  - 每日跑一次：拉取当日文章列表 → 取正文 → 按「公众号 prompt」做摘要/要点 → 写入内容池。  
- **推送**：类型 2，每日一封聚合（可与门户网站合并为同一封）。

### 3.3 Twitter / HN / Kickstarter（需求 3）

- **HN**：已有 RSS，可视为「门户」之一，纳入每日聚合；若需「按话题」则放在「固定话题」里用关键词/标签筛。
- **Twitter**：需 API 或爬虫，可能需登录；建议单独做「数据源适配器」，输出统一条目格式进内容池；话题筛选可在适配器内按关键词/hashtag，或交给「固定话题」层。
- **Kickstarter**：同理，独立数据源适配器 + 统一条目格式；若需登录再在适配器内做鉴权。
- 三者都可作为「固定话题」的来源（见下）。

### 3.4 投资板块（需求 4）

- **数据**：行情/公告/研报/大 V 文章等；大 V 可与「公众号」或「门户」复用一部分数据源。  
- **逻辑**：  
  - 按「投资」拉取/筛选 → 入池 → 用「投资专用 prompt」生成当日信号+建议。  
  - 分区：港股/A 股 vs 美股；两档推送时间（中午 / 23:00）。  
- **推送**：类型 3，两档独立任务，各发一封（或两封合并为「投资日报」但内容分块）。

### 3.5 固定话题（需求 5）

- **输入**：在 config 里配置「话题列表」（关键词或简单规则）。  
- **来源**：热榜、RSS、公众号、门户、播客摘要、Twitter、投资等已入池内容，统一按「话题」检索/过滤。  
- **逻辑**：  
  - 定时或跟随某次抓取：从内容池检索 → 按话题聚合 → 用「话题 prompt」整理 → 输出。  
- **推送**：可归入类型 2（每日一封）或单独一个「话题日报」类型，看产品形态再定。

### 3.6 三种邮件类型（其他）

- **类型 1（播客）**：  
  - 触发：某播客 feed 有新条目且 ASR+摘要 完成。  
  - 推送记录：可按 `report_type=podcast_{feed_id}` 或按条记录，避免重复发。  

- **类型 2（公众号+门户）**：  
  - 触发：每日一次定时任务。  
  - 推送记录：沿用或扩展 `once_per_day`，用 `report_type=daily_digest`。  

- **类型 3（投资）**：  
  - 触发：两档 cron — 中午（港股/A 股）、23:00（美股）。  
  - 推送记录：`report_type=investment_cn` / `investment_us`，每档每天最多 1 次。  

现有 `PushRecordManager` / `has_pushed_today` / `record_push(report_type)` 可扩展为「按 report_type 分别判断是否已推送」，而不是全局一次。

## 四、技术选型与风险（简要）

| 项目 | 建议 | 风险/备注 |
|------|------|-----------|
| ASR | 优先用现有 AI 提供方的 ASR API，或 Whisper API | 长音频成本与时长限制；可先做「单集长度上限」 |
| 公众号 | 先接第三方 RSS 或 1 个明确源 | 爬虫易挂；合规注意 |
| Twitter | API 或只读爬虫，需鉴权 | 限流、封号风险 |
| 投资数据 | 先行情/公开公告+已有 RSS/公众号 | 「建议」仅作参考，合规表述要明确 |
| 存储 | 内容池可新表或新 DB，与现有 news/rss 并存 | 需设计「来源 + 日期 + 已推送」等字段便于去重与检索 |

## 五、分批实现顺序建议

1. **Phase 1 — 推送类型与调度**  
   - 在配置和推送层区分三种邮件类型及时间策略。  
   - 扩展 `report_type`，按类型记录推送，不破坏现有热榜/RSS 推送。

2. **Phase 2 — 播客 pipeline**  
   - 播客 RSS 配置 → 新条目检测 → 音频下载 → ASR → prompt 摘要 → 入池 + 类型 1 推送。  
   - 先支持 1 个播客源、1 种 ASR，验证端到端。

3. **Phase 3 — 公众号 + 每日聚合**  
   - 接 1 个公众号数据源（RSS 或其它）→ 每日拉取 → prompt 摘要 → 入池 → 类型 2 推送。  
   - 与现有「门户/RSS 日报」合并或并列。

4. **Phase 4 — 投资板块**  
   - 投资数据源 + 投资 prompt → 港股/A 股 vs 美股两档 → 类型 3 推送。

5. **Phase 5 — Twitter / Kickstarter + 固定话题**  
   - 数据源适配器（含登录若需要）→ 入池。  
   - 固定话题：配置话题列表 + 从池中检索 + 话题 prompt → 整理输出。

6. **Phase 6 — 优化与扩展**  
   - 多播客、多公众号、多模型（如 siliconflow 仅改 config）、限流与重试等。

## 六、需要你拍板的点

1. **播客 ASR**：是否已有偏好的 ASR 服务（Whisper / 国内云 ASR / 当前 AI 厂商）？是否接受「单集长度上限」或「只处理前 N 分钟」以控制成本？  
2. **公众号**：是否已有可用的第三方 RSS 或 API（如 RSSHub、即刻、其它），还是希望我们只留「配置位」后续再接？  
3. **投资**：「投资建议」的合规表述是否需固定免责文案？数据源是否已有名单（例如某几个公众号/站点）？  
4. **Twitter/Kickstarter**：是否必须首版就上，还是可以排在「固定话题」之后、只做数据源占位？  
5. **固定话题**：首版是「仅关键词匹配」即可，还是希望有「语义/向量检索」？

你确认或调整上述几点后，可以按「Phase 1 → 2 → …」拆成具体开发任务（含配置文件字段、表结构、接口），再落到代码实现。若你希望我先细化某一 Phase 的接口与配置设计，也可以直接说从哪一块开始。
