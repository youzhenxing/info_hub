## 📝 更新内容

### 新增章节

在 CLAUDE.md 中添加了完整的"生产环境部署规范"章节，包含以下内容：

#### 1. 强制要求

```
⚠️ 生产环境部署必须使用标准部署流程，禁止使用手动部署。
```

**核心原则**:
- ✅ 生产环境 → 必须使用 `deploy/deploy.sh`
- ❌ 禁止直接 docker build + docker run
- ⚠️ 手动部署仅限开发测试

#### 2. 标准部署流程（4步）

**步骤1: 代码提交**
```bash
git add <modified_files>
git commit  # Pre-commit hook验证
```

**步骤2: 执行部署脚本**
```bash
cd /home/zxy/Documents/code/TrendRadar/deploy
./deploy.sh
```

自动完成：
- ✅ 6项部署前检查
- ✅ 版本管理
- ✅ Docker镜像构建（带版本标签）
- ✅ 文件同步到生产环境
- ✅ 配置校验
- ✅ **自动发送部署通知邮件** ← 重点
- ✅ 创建部署历史记录

**步骤3: 切换版本**
```bash
trend update v{VERSION}
```

**步骤4: 验证部署**
```bash
docker ps | grep trendradar-prod
docker logs trendradar-prod --tail 50
```

#### 3. 标准部署的优势

1. **版本管理**: 保存所有历史版本，支持快速回滚
2. **环境隔离**: 生产配置独立，不受开发环境影响
3. **完整检查**: 6项部署前检查，确保部署安全
4. **自动通知**: 部署成功后自动发送邮件通知
5. **可追溯**: 完整的部署历史和变更记录
6. **快速回滚**: 支持一键回滚到任意历史版本

#### 4. 手动部署的适用场景

**仅限以下场景可以使用手动部署**：
- ✅ 开发测试阶段的快速迭代
- ✅ 实验性功能的临时部署
- ✅ 本地环境调试

**明确禁止**：
- ❌ 生产环境部署
- ❌ 正式版本发布
- ❌ 团队协作项目

#### 5. 部署检查清单

部署前必须确认：
- [ ] 代码已提交到 Git（有 Commit ID）
- [ ] 所有测试已通过
- [ ] 使用 `deploy/deploy.sh` 脚本部署
- [ ] 版本号已更新（如有需要）
- [ ] 部署通知邮件已收到
- [ ] 容器运行状态正常
- [ ] 功能验证通过

#### 6. 常见错误示例

**❌ 错误示例**:
```bash
# 错误1: 直接手动构建部署
cd docker && ./build-local.sh
docker stop trendradar-prod && docker rm trendradar-prod
docker run -d ...  # ❌ 缺少版本管理、检查、通知

# 错误2: 跳过部署脚本
docker build -t trendradad:latest .
docker run ...  # ❌ 无版本号，无部署记录
```

**✅ 正确示例**:
```bash
# 正确: 使用标准部署流程
git add trendradar/podcast/processor.py
git commit -m "feat(podcast): 添加新功能"

cd deploy
./deploy.sh  # ✅ 完整的部署流程

trend update v5.26.0  # ✅ 切换到新版本
```
