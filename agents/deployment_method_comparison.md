## 📊 部署流程对比

### 标准部署流程（使用 deploy.sh）

```
1. 部署前检查（pre-deploy-check.sh）
   ├─ 检查源文件（entrypoint.sh, run_investment.py等）
   ├─ 检查脚本路径配置
   ├─ 检查 Docker Compose 配置
   ├─ 检查执行权限
   ├─ 检查模块状态
   └─ 检查生产环境配置

2. 版本管理
   ├─ 读取版本号（从 version 文件）
   ├─ 检查版本是否已存在
   └─ 确认是否覆盖现有版本

3. 构建 Docker 镜像
   ├─ 构建主镜像: trendradar:v{VERSION}
   └─ 构建 MCP 镜像: trendradar-mcp:v{VERSION}

4. 创建版本目录
   ├─ 创建 releases/v{VERSION}/
   └─ 复制核心代码、配置、依赖

5. 文件同步到生产环境
   ├─ 复制共享脚本（run_investment.py, entrypoint.sh等）
   ├─ 复制 shared 模块（email_renderer, 邮件模板）
   ├─ 复制 prompts 提示词
   ├─ 同步 .env 配置
   └─ 校验必选配置项

6. 生成 Docker Compose 配置
   ├─ 生成 docker-compose.yml
   ├─ 配置容器名、重启策略
   ├─ 配置端口、挂载卷、环境变量
   └─ 配置 MCP 服务

7. 创建版本记录
   ├─ 读取 git commit 信息
   ├─ 创建详细版本记录
   ├─ 更新 manifest.yaml
   └─ 添加部署历史

8. 发送部署通知邮件 ✅
   ├─ 检查播客源、投资数据源
   ├─ 检查数据库、配置
   ├─ 检查监控服务、Wewe-RSS
   └─ 发送完整状态邮件

9. 完成（但不自动重启容器）
   └─ 需要手动执行: trend update v{VERSION}
```

### 手动部署流程（本次使用）

```
1. 代码修改
   ├─ 修改 trendradar/podcast/processor.py
   ├─ 修改 trendradar/core/loader.py
   └─ 修改 config/config.yaml

2. Git 提交
   ├─ git add (手动添加文件)
   ├─ git commit (Pre-commit hook验证)
   └─ Commit ID: 911513e2

3. Docker 镜像构建
   └─ docker/build-local.sh
       └─ 生成: trendradar:local

4. 容器重启
   ├─ docker stop trendradar-prod
   ├─ docker rm trendradar-prod
   └─ docker run (新镜像)

5. 功能验证
   └─ 手动测试新逻辑

6. 手动发送邮件通知（补救）
   └─ python3 deploy/send_deploy_notification.py
```

## 🔍 关键差异分析

### 1. 版本管理

| 方面 | 标准部署 | 手动部署 |
|------|----------|----------|
| **版本号** | 从 version 文件读取，语义化版本（如 v5.26.0） | 无版本号，使用 :latest 或 :local |
| **版本历史** | ✅ 保存到 releases/ 目录和 manifest.yaml | ❌ 无版本历史 |
| **回滚能力** | ✅ 可以快速回滚到任意历史版本 | ❌ 需要重新构建镜像 |
| **版本记录** | ✅ 包含 git commit 信息、部署时间、变更说明 | ❌ 无记录 |

**示例**:
```bash
# 标准部署
/home/zxy/Documents/install/trendradar/releases/
├── v5.25.0/
├── v5.25.1/
├── v5.26.0/
└── manifest.yaml  # 记录所有版本

# 手动部署
# 无版本目录，只有镜像
docker images | grep trendradad
# trendradad:local  (无版本信息)
```

### 2. 文件同步

| 文件类型 | 标准部署 | 手动部署 |
|----------|----------|----------|
| **共享脚本** | ✅ 复制到 $PROD_BASE/shared/ | ❌ 挂载开发目录 |
| **Prompts** | ✅ 复制到生产环境独立目录 | ❌ 挂载开发目录 |
| **邮件模板** | ✅ 复制到生产环境独立目录 | ❌ 挂载开发目录 |
| **.env配置** | ✅ 同步并校验必选配置 | ❌ 使用开发目录配置 |

**影响**:
- **标准部署**: 生产环境独立，开发环境改动不影响生产
- **手动部署**: 开发环境改动直接影响生产（风险更高）

---

### 3. 配置管理

| 方面 | 标准部署 | 手动部署 |
|------|----------|----------|
| **配置来源** | 生产环境独立配置（$PROD_BASE/shared/config） | 开发环境配置（挂载） |
| **环境变量** | 生产环境 .env 文件 | Docker run -e 参数 |
| **配置校验** | ✅ 部署前检查必选配置项 | ❌ 无校验 |
| **配置隔离** | ✅ 开发和生产完全隔离 | ❌ 共享配置文件 |

**风险点**:
```bash
# 手动部署
docker run -v /home/zxy/Documents/code/TrendRadar/config:/app/config:ro ...
# ⚠️ 如果开发环境配置错误，直接影响生产

# 标准部署
volumes:
  - $PROD_BASE/shared/config:/app/config:ro
# ✅ 生产环境配置独立，不受开发影响
```

## 🎯 适用场景分析

### 标准部署适用场景

✅ **推荐使用标准部署的情况**:
1. **生产环境部署**: 正式发布到生产环境
2. **多版本管理**: 需要保留多个历史版本
3. **快速回滚**: 可能需要回滚到之前的版本
4. **环境隔离**: 开发和生产环境需要完全隔离
5. **团队协作**: 多人协作，需要标准化的部署流程
6. **审计需求**: 需要完整的部署历史和变更记录

❌ **不适合标准部署的情况**:
1. 快速测试开发中的功能
2. 本地调试和实验性修改
3. 不需要版本管理的简单部署

### 手动部署适用场景

✅ **适合手动部署的情况**:
1. **快速迭代**: 开发阶段快速测试修改
2. **实验性功能**: 临时性、实验性的修改
3. **简单部署**: 不需要版本管理的单容器部署
4. **紧急修复**: 快速修复紧急问题
5. **本地开发**: 本地环境测试和调试

❌ **不适合手动部署的情况**:
1. 正式生产环境部署
2. 需要保留多个版本
3. 需要快速回滚能力
4. 团队协作的生产系统

## 🚀 推荐部署策略

### 开发阶段

```bash
# 使用手动部署（快速迭代）
1. 修改代码
2. git commit
3. cd docker && ./build-local.sh
4. docker stop trendradar-prod && docker rm trendradar-prod
5. docker run -d --name trendradar-prod ...
6. 手动测试验证
```

### 生产部署

```bash
# 使用标准部署（规范管理）
1. 修改代码并测试
2. git commit
3. cd deploy && ./deploy.sh
   - 自动检查
   - 自动构建镜像
   - 自动同步文件
   - 自动创建版本记录
   - 自动发送邮件通知
4. trend update v{VERSION}  # 切换到新版本
5. 验证新版本运行状态
```

### 混合策略

```bash
# 开发测试: 手动部署
# 正式发布: 标准部署

# 本地测试
docker/build-local.sh + docker run

# 确认无误后正式发布
cd deploy && ./deploy.sh
```

## 🔧 改进建议

### 手动部署的改进

如果需要继续使用手动部署，建议增加以下检查步骤：

```bash
#!/bin/bash
# 简化的部署前检查脚本

echo "🔍 部署前快速检查..."

# 1. 检查配置文件语法
echo "检查配置文件..."
python3 -c "import yaml; yaml.safe_load(open('config/config.yaml'))" || exit 1

# 2. 检查Python语法
echo "检查Python语法..."
python3 -m py_compile trendradar/podcast/processor.py || exit 1

# 3. 检查容器状态
echo "检查容器..."
docker ps | grep trendradar-prod || echo "⚠️ 容器未运行"

# 4. 构建镜像
echo "构建镜像..."
cd docker && ./build-local.sh

# 5. 重启容器
echo "重启容器..."
docker stop trendradar-prod
docker rm trendradar-prod
docker run -d --name trendradar-prod ...

# 6. 发送通知
echo "发送部署通知..."
python3 deploy/send_deploy_notification.py "manual-$(date +%Y%m%d-%H%M%S)"

echo "✅ 部署完成"
```

### 标准部署的改进

可以考虑增加容器自动重启选项：

```bash
# 在 deploy.sh 末尾添加
echo "是否立即重启容器到新版本？(y/N)"
read -r confirm
if [[ "$confirm" =~ ^[Yy]$ ]]; then
    trend update v${VERSION}
    echo "✅ 容器已重启到版本 v${VERSION}"
else
    echo "⏸️  容器未重启，请手动执行: trend update v${VERSION}"
fi
```

## 🎯 结论

### 标准部署的优势

1. **规范性**: 标准化的流程，减少人为错误
2. **可追溯**: 完整的版本历史和部署记录
3. **安全性**: 环境隔离，配置校验
4. **可维护**: 支持快速回滚和版本切换
5. **自动化**: 自动检查、自动通知

### 手动部署的优势

1. **快速**: 简化流程，部署速度快
2. **灵活**: 可以自由定制部署步骤
3. **简单**: 不需要理解复杂的部署脚本
4. **适合**: 开发测试和快速迭代

### 推荐做法

```
开发阶段 → 手动部署（快速迭代）
        ↓
    测试验证
        ↓
生产发布 → 标准部署（规范管理）
```
