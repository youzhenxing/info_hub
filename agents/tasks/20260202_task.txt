### 2. 配置统一化重构

#### 2.1 问题发现
用户询问：".env 和 yaml 配置文件的关系"

**发现**:
- 微信模块使用独立的 `.env` 加载机制（`load_dotenv()`）
- 其他模块使用统一的 `config/system.yaml`
- 配置机制不一致

#### 2.2 重构目标
统一所有模块配置管理到 `config/system.yaml`

#### 2.3 实施步骤

**步骤 1**: 移除独立的 .env 加载
```python
# wechat/src/config_loader.py
# 移除前：
from dotenv import load_dotenv
load_dotenv()

# 移除后：
# （已删除）
```

**步骤 2**: 清空 .env 文件
```bash
# wechat/.env
# 已弃用：配置已迁移到统一的 config/system.yaml
```

**步骤 3**: 更新 wechat/config.yaml
```yaml
# 敏感信息改为空，从 system.yaml 读取
email:
  from: ""
  password: ""
  to: ""

ai:
  api_key: ""
  api_base: ""
```

**步骤 4**: 增强 ConfigLoader
```python
def _load_config(self):
    """加载配置文件（支持 system.yaml 和本地 config.yaml）"""
    # 1. 加载 system.yaml
    system_config = load_system_config()

    # 2. 加载本地 config.yaml
    local_config = load_local_config()

    # 3. 合并配置（system.yaml 覆盖本地配置）
    self._config = self._merge_configs(local_config, system_config)
```

**步骤 5**: 修正配置优先级
```
1. 环境变量（最高优先级）
2. config/system.yaml（统一配置）
3. wechat/config.yaml（本地默认）
```

## 💡 经验总结

### 技术洞察
1. **配置优先级**: 环境变量 > 统一配置 > 本地配置
2. **向后兼容**: 保留本地配置作为默认值很重要
3. **安全意识**: .env 文件不应该提交到版本控制

### 开发流程
1. **问题定位**: 通过日志和测试快速定位问题
2. **分步修复**: 一次解决一个问题，避免引入新问题
3. **充分测试**: 重构后必须进行完整测试

### 部署实践
1. **版本管理**: 使用版本目录 + 软链接机制
2. **灰度发布**: 先部署再切换，支持快速回滚
3. **通知机制**: 邮件通知确保团队同步
